# AGENTS.md

本文件规定 AI 代理在本仓库中的工作方式。所有代理必须优先遵守本文件；当本文件与用户的最新明确指令冲突时，以用户最新指令为准，并在执行前说明影响。

## 项目定位

本项目是 Xinzhi Lin 的个人静态作品集网站，面向 GitHub Pages 直接部署。仓库采用原生 Web 技术：

- HTML5
- CSS3
- Vanilla JavaScript
- 静态资源文件

除非用户明确批准，严禁引入 React、Vue、Sass、构建工具、静态站点生成器、后端服务或任何会改变部署模型的新依赖。

## 资料优先级

开始实现前，必须先查证仓库内已有信息，避免凭印象推断。

优先参考以下项目资料：

- `Knowledge Items/site-structure.md`
- `Knowledge Items/site-content.md`
- `Knowledge Items/design-system.md`
- `Knowledge Items/coding-rules.md`
- 现有 HTML、CSS、JavaScript 文件

能通过代码库或项目文档确认的问题，必须先自行查证。只有在查证后仍无法确定，且该问题会影响需求、方案、边界条件或架构决策时，才向用户提问。

## 沟通规则

- 与用户保持中文交流。
- 严禁使用 Emoji。
- 需求、方案、边界条件或架构决策不清时，严禁直接写代码。
- 与用户沟通需求、方案、边界条件和架构决策时，必须使用 `grill-with-docs` 技能。
- 需要用户拍板时，尽量使用“选择题 + 选项 + 简要分析 + 建议”的格式，并逐步推进到需求具象化。
- 若问题能通过阅读代码、文档或 Git 历史确认，应先自行确认，再把结论反馈给用户。

## 实现原则

- 坚持最小正确实现。
- 只实现确实需要的内容，严禁拓展无关功能。
- 不主动重构无关文件。
- 不主动改变现有信息架构、视觉风格、交互方式或部署方式。
- 修改范围应尽量贴近用户请求所涉及的页面、样式或脚本。
- 新增抽象、工具函数或共享样式前，必须确认它能减少真实重复或匹配现有模式。
- 对已有内容保持克制，除非用户明确要求，不重写文案、不替换资产、不新增页面。

## 编码与文件规则

- 读写文件必须使用 UTF-8 编码，防止中文乱码。
- 保持文件格式与现有代码风格一致。
- HTML、CSS、JavaScript 优先沿用现有结构、命名和变量。
- 避免引入非必要的 Unicode 符号；文案中严禁 Emoji。
- 不使用会破坏静态部署的路径、运行时依赖或本地专用配置。
- 外部链接、资源路径和相对路径必须与 GitHub Pages 静态部署兼容。

## 设计与交互边界

项目视觉基调是现代、极简、清晰、留白充足的个人学术与作品展示站点。实现时应延续现有设计系统：

- 白色或浅灰背景
- 近黑主文本与中灰辅助文本
- 蓝色作为主要行动色
- 克制使用红、黄、绿品牌辅助色
- 清晰的排版层级
- 轻量 hover 状态与滚动 reveal 动效

除非用户明确要求，严禁新增复杂动效、营销式组件、沉重视觉装饰或与现有 Google aesthetic 不一致的风格。

## 博客写作规则

撰写或修改博客时，语言应严谨但不死板，学术但不生硬。

- 避免连续堆叠长句。
- 优先使用清晰、直接的肯定表达。
- 尽可能避免使用“不是……而是……”句式。
- 保持观点、论证和例子之间的逻辑清楚。
- 避免空泛抒情、夸张表达和未经证实的判断。
- 保持与站点整体气质一致：克制、清楚、可信。

## Git 与验收

- 不在用户未批准的情况下创建提交。
- 当用户明确认可某个页面或主要功能完成，并要求或允许提交时，按 `Knowledge Items/coding-rules.md` 的约定创建简洁、专业的提交信息。
- 修改后应根据变更类型进行适当检查。静态页面修改至少应检查目标文件是否可读、路径是否正确、明显语法是否合理。
- 若未能运行检查，必须在最终回复中说明原因。

## 禁止事项

- 禁止实现用户未要求的功能。
- 禁止因个人判断扩展需求范围。
- 禁止在需求不清时直接编码。
- 禁止引入无关依赖或框架。
- 禁止使用 Emoji。
- 禁止未经确认改动项目部署模型。
- 禁止覆盖、删除或回退用户已有修改，除非用户明确要求。
